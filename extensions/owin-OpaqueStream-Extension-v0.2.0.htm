1.    Introduction

This document outlines the recommended patterns for using opaque streams through the OWIN interface. It depends on OWIN version 0.12.0 (but may work with other versions), and uses the extension key mechanics as outlined in CommonKeys.html.

2.    Definitions

Opaque Streams or Opaque Mode - Some servers allow a request to take direct ownership of the underlying connection after the initial HTTP handshake. This gives the application access to a bidirectional communication channel and the application is then responsible for employing its own protocol (e.g.  WebSockets) to communicate with the client.

3.    Scenarios

At startup the server SHOULD advertise via keys in the IAppBuilder Properties dictionary what capabilities it supports.  See the recommended Keys below.  This advertisement helps the application build a pipeline to fit its needs.

3.1.  The application needs opaque stream access and the server supports it.

·         When the server identifies a request that can be upgraded to an opaque, it SHOULD indicate this to the application by adding the appropriate key and value to the Environment dictionary (see Keys). 

·         Applications that see this key and want to complete the upgrade can indicate this to the server by returning a 101 status code and adding an opaque stream body delegate to the response Properties dictionary (see Keys, Delegate Signature & Usage).  

o   The normal response body delegate field SHOULD be set to null.

·         The server then performs the upgrade and invokes the opaque stream body delegate.

4. Keys

The following keys and values are defined according to the guidelines provided in CommonKeys.html.  Key names and string values MUST be compared as Ordinal, case sensitive.

Type

opaque.Support

string

“BodyFunc” Used in the IAppBuilder Properties dictionary to indicate general opaque stream support.  Used in the request Environment dictionary to indicate an opaque stream compatible request.

opaque.BodyFunc

Func<...>

Used in the response Properties dictionary to request that the server upgrade to an opaque stream and invoke this delegate.  See Delegate Signature & Usage.

 

5. Delegate Signature & Usage

The opaque stream body delegate signature is as follows:

using OpaqueStreamAction =

        Func

        <

            Stream, // Incoming (CanRead)

            Stream, // Outgoing (CanWrite)           

            Task // Complete

        >;

 

The application provides this delegate in the response Properties dictionary.  The server invokes it to provide the application with Streams for incoming and outgoing data.  The consumer then operates their own protocol (e.g. WebSockets) over these streams.  On completion (success or fail), the application MUST complete or fail the returned Task.  The consumer SHOULD NOT close the given streams, the provider (the server) owns these and is responsible for any cleanup on completion.  The application SHOULD NOT continue to access the Streams once it completes the returned task.
