1.  Overview

This document contains guidelines for adding functionality to an OWIN compatible host or application via additional keys in the request Environment and response Properties IDictionary<string, object> collections.  This includes naming conventions, value conventions, and a list of know common keys and their semantics.

These guidelines and keys are provided independently of the OWIN specification as the list of common keys below is expected to grow independently of the OWIN standard.

2. Key usage guidelines

The request Environment and response Properties IDictionary collections can be used to extend the OWIN interface with additional functionality. 

·         All keys must be compared using StringComparer.OrdinalIgnoreCase.

·         All keys not defined in the OWIN standard are considered extension keys and are strictly optional.

·         Implementers should avoid providing keys with null values, but must be tolerant of null values when retrieving data.

·         Implementers should clearly document what keys their component will populate, under what conditions, and the types and semantics of the associated values.

3. Naming conventions

The Environment and Properties dictionaries are open, mutable collections that can be used to store arbitrary data.  To avoid collisions between modules, the following conventions should be honored.

·         Keys defined directly in the OWIN standard use the “owin.” prefix.  No additional keys should be added that use this prefix.

·         A number of common keys are listed later in this document.  These keys use general prefixes such as host.* to indicate that this is data that may be provided by many different implementations.  Implementers should favor using such shared keys rather than defining their own.

·         Keys that refer to a technology group should be prefixed with a name that identifies that technology. E.g. Ssl.* or WebSocket.*.

·         Keys that expose functionality specific to a single implementation should use a prefix that clearly identifies that implementation.  E.g. IIS.*, HttpListener.*, WebApi.*, MyMiddleware.*.

·         Implementers should consider including a technology specific version key that clearly identifies not only the version of the underlying component, but also the version of the OWIN specific wrapper of that component.  E.g. HttpListener.Version: .NET 4.5, OWIN wrapper 1.0.1.  This is intended to help consumers cross reference documentation with implementations, facilitating both development and debugging.

o   Where the version includes the .NET framework version, specify the version compiled against, not the version you are currently running.  E.g. if you targeted .NET 4.0 and are running on .NET 4.5, specify 4.0.  The version you are currently running on is readily available in Environment.Version.

·         The fully qualified name of a type may be used as the key, but only if it is reasonable to assume that there will only ever be one instance of this type per request.  E.g. System.Net.HttpListenerContext is guaranteed to be unique per request, whereas System.Security.Cryptography.X509Certificates.X509Certificate identifies a specific type but it is ambiguous if the value represents the client cert, the server cert, or a cert for some other purpose.

4. Value conventions

·         The type of each value must be clearly documented by the implementer.

·         Values may be of a type derived from the type specified in documentation, but the documentation should state under what conditions it is reasonable to expect such types to be available.  Consumers may attempt to cast to more specific derived types (safely using “is” or “as” rather than an explicit cast), but should fall back to the documented base type wherever possible.

·         Null values should be treated the same as if the key were not present.

·         Avoid overloading a single key with multiple types (other than derived types as discussed above).  If there are multiple possible representations, use separate keys to clearly identify them.

·         Avoid using types that are likely to change signatures between versions. 

·         Prefer base types (e.g. int, string, etc.) or common types in the .NET framework that have existed for multiple releases (e.g. Stream).

·         Values are assumed to be alive and accurate only for the lifetime of the request/response.  Consumers should not retain or attempt to use values after a request has completed or failed.  Where a value advertises a specific capability, that capability is only assumed to be available for that request instance and should be re-verified on future requests.

·         Where any specific value cleanup is required (e.g. Dispose), such cleanup is the responsibility of the component that originally added the value.  Consider registering with the request completion CancellationToken as a definitive means of identifying the end of the request scope.
